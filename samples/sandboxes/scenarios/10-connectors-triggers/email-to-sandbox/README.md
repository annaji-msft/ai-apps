# Scenario — email → sandbox → AI-composed reply

**Round-trip Office 365 → sandbox → Office 365.** A connector-gateway
trigger fires on a new "Feedback" email, the sandbox hands the body
to the pre-installed **GitHub Copilot CLI**, and a personalized
acknowledgment is sent back through the **same** Office 365
connection — one OAuth consent, both directions.

```
┌──────────────┐  new email  ┌────────────────────┐
│  Your Inbox  │ ──────────▶ │  Connector Gateway │
│  (O365)      │  subject:   │  (OnNewEmailV3)    │
│              │  Feedback…  │                    │
└──────────────┘             └─────────┬──────────┘
                                       │ POST /webhook  (MSI bearer)
                                       ▼
                ┌──────────────────────────────────────┐
                │ Sandbox (copilot disk image)         │
                │   listener :5000 (stdlib http)       │
                │     ├── ack 200 immediately          │
                │     └── worker:  copilot -p <prompt> │
                │                  → POST /v2/Mail     │
                └──────────────────────┬───────────────┘
                                       │ Authorization injected by
                                       │ platform proxy (SG MI via
                                       │ gatewayConnections wiring)
                                       ▼
                            ┌────────────────────────┐
                            │  Connector Gateway     │
                            │  (SendMailV2)          │
                            └─────────┬──────────────┘
                                      ▼
                            ┌────────────────────────┐
                            │ Triage Inbox           │
                            │ "Auto-ack: received…"  │
                            └────────────────────────┘
```

## What you get

For every email whose subject contains **Feedback**, you receive a
reply titled `Auto-ack: received your message` containing a warm,
AI-composed acknowledgment that references the points you raised,
plus a footer noting the reply was generated by the Copilot CLI.
The prompt template is `_COPILOT_PROMPT_TEMPLATE` in
[`app/server.py`](app/server.py) — tweak it for persona, length,
language, or structured output.

## Prerequisites

```bash
# 1. sandboxes baseline (resource group + sandbox group)
python ../../../setup/python/setup.py     # or .../cli/setup.sh

# 2. connector-gateway + Office 365 connection + both access policies
python ../setup/python/setup.py            # or .../cli/setup.sh
```

The scenario setup walks you through the one-time OAuth consent for
the Office 365 connection and provisions a system-assigned managed
identity on the sandbox group with a matching `sandbox-acl` access
policy, so the listener can call the same connection's `/v2/Mail`
endpoint.

You also need a **GitHub token** for the Copilot CLI. The driver
picks one up in this order — no manual config required if you have
`gh auth login` done locally:

1. `COPILOT_GITHUB_TOKEN` / `GH_TOKEN` / `GITHUB_TOKEN` env vars.
2. `gh auth token` from the [GitHub CLI](https://cli.github.com/).
3. An interactive paste-and-hide prompt at run time.

Accepted: OAuth tokens (`ghu_`, `gho_`) and fine-grained PATs with
the **Copilot Requests** permission. Classic `ghp_` tokens are
**not** supported. The token is injected as an env var into the
listener process (the headless-auth path documented by
`copilot help environment`) and is destroyed with the sandbox.

Optional:

```
TRIAGE_RECIPIENT=triage@contoso.com   # defaults to ACA_USER_EMAIL
```

## What the script demonstrates

1. **One connection, both directions.** The same Office 365
   connection that fires `OnNewEmailV3` is reused for `SendMailV2`.
2. **MSI callback auth.** The trigger sets
   `notificationDetails.authentication = { type: ManagedServiceIdentity,
   audience: https://auth.adcproxy.io/ }`; the gateway's POST
   carries a Bearer token validated by the sandbox proxy against the
   port's `entraId.objectIds`.
3. **AI inside the sandbox.** The pre-installed `copilot` CLI
   composes the reply with one `subprocess.run` per email,
   serialized by a `threading.Semaphore` so bursts can't fork-bomb
   the sandbox.
4. **Platform-injected outbound auth (declarative).** The sandbox
   calls `{RUNTIME_URL}/v2/Mail` with no `Authorization` header. The
   sandbox group declares the connection in its `gatewayConnections[]`
   (`SystemAssignedManagedIdentity` auth, runtime URL from the PUT
   response), and `run.py` creates each sandbox with
   `gatewayConnections=[{ resourceId }]` so the platform proxy injects
   the Bearer token for you. No egress Transform rule needed.
5. **Deny-by-default egress** with host-Allow for the GitHub CLI
   host families only (`github.com`, `*.github.com`,
   `*.githubusercontent.com`, `gh.io`, `*.github.io`,
   `*.githubcopilot.com`). The connector runtime host is reached via
   the platform-mediated path above, so it doesn't need an egress
   Allow entry.
6. **Loop-safe reply subject.** The outbound subject is free of the
   word "Feedback", and the prompt + a defensive `re.sub` keep it
   out of the body, so replies don't retrigger.
7. **Fire-and-forget listener.** The handler returns 200 immediately
   and processes in a worker thread, so the connector doesn't retry
   while `copilot` is composing.
8. **Cleanup order: trigger → port → sandbox.** The gateway,
   connection, and access policies are kept for re-runs.

## Notes

- **Subject filter.** `SUBJECT_FILTER` (top of `run.py` / `run.sh`)
  defaults to `Feedback`. `REPLY_SUBJECT_PREFIX` in `app/server.py`
  MUST NOT contain that word or you get an infinite self-trigger.
- **Latency.** Expect a ~1 minute lag from "send" to "reply
  arrives" — `metadata.recurrenceFrequency=Minute` /
  `recurrenceInterval=1` is already the floor for this connector.

## Where to take this next

- **Structured output.** Have Copilot emit JSON (sentiment, category,
  urgency) and route based on it.
- **Multi-turn context.** Use `--name`/`--resume` keyed on
  `conversationId`/`internetMessageId` for thread-aware replies.
- **Different model or provider.** Pass `--model gpt-5.2`, or set
  `COPILOT_PROVIDER_BASE_URL` for an OpenAI-compatible endpoint
  (see `copilot help environment`).
- **Durable queue.** Replace the in-memory worker thread with Azure
  Storage Queues / Service Bus so in-flight work survives a crash.
- **Outbox de-dupe.** Track processed `internetMessageId`s to avoid
  double-replies on connector retries.
